<objective>
Close the two deferred `weather` code-review warnings from `10-REVIEW.md` (WR-01,
WR-02) that the developer has now chosen to fix. Both live entirely inside the
`weather` feature shipped by 10-05.

- **WR-01** — `--forecast` claims a "bounded 7-day" span everywhere but never
  enforces it: `build_forecast_url` omits Open-Meteo's `forecast_days` param (so the
  span is the API's drift-prone runtime default), and `build_day_forecasts` bounds
  only *mismatched* lengths (`n > 0` + equal), never the *size* (`n <= 7`). A 16-day
  (or anomalously large) response is accepted, rendered under the hard-coded
  `"  7-day forecast:"` header, and serialized verbatim into the `--json` `forecast`
  array. Fix belt-and-suspenders: (a) pin the span server-side (`&forecast_days=7`),
  (b) enforce `n <= 7` defensively.
- **WR-02** — the cache key + geocode use the raw, un-normalized location string, so
  `"London"` / `" London "` / `"london"` hash to different cache keys and each pays a
  fresh network + geocode round-trip. Fix: normalize once (`.trim()`) right after
  `resolve_location`, and lowercase CITY names (not `lat,lon` pairs) before hashing
  into the cache key.

Purpose: make the shipped "bounded 7-day forecast" invariant real (both docs and
runtime), and make the new response cache actually share entries across incidental
whitespace/case — the miss-rate gap the caching feature was meant to eliminate.

Output: an amended `weather` module (URL pin + defensive bound + location
normalization + de-overclaimed doc comments) and regression tests.
</objective>
