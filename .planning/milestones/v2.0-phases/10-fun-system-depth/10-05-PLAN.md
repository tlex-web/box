<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Open-Meteo HTTP response → process | remote JSON deserialized via `fetch()`; must be bounded + shape-validated |
| user-controlled location → cache key → filesystem | the location string forms the cache key; the hashing lives in core::cache (10-04) — this plan must pass the logical key, never a raw path |
| cache payload on disk → render | a cached payload may be stale/malformed; core::cache returns None on any such case (a MISS) |
| stderr/cache activity → --json stdout | machine stdout must stay a single clean document |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-10-05-HTTP | Denial of Service | forecast response deserialize | mitigate | The daily forecast is a fixed 7-day span deserialized into a struct with typed `Vec` fields (shape-bounded, not an open-ended stream); a short/mismatched daily array is treated as an error, never a panic. `fetch()`'s ureq status-as-error split is reused verbatim (non-2xx → exit 1, offline → graceful exit 1). |
| T-10-05-CACHE-KEY | Tampering | cache key from location | mitigate | The logical key (location,units,forecast) is passed to `core::cache`, which blake3-hashes it into the filename (10-04, T-10-04-TRAVERSAL) — this plan NEVER writes a filename from the raw location. |
| T-10-05-JSON-PURITY | Info disclosure | cache/stderr under --json | mitigate | The resolved-location echo stays stderr-only; cache reads/writes emit nothing to stdout; the existing `json_purity` harness asserts stdout is exactly one JSON value with no ANSI/BOM. |
| T-10-05-CACHE-POISON | Tampering | attacker-touched cache file | mitigate | A malformed/attacker-written cache entry is a MISS (core::cache returns None) → a fresh fetch; the command never trusts an unvalidated cached shape (delegated to 10-04's tolerant read). |
| T-10-05-LOC-INJ | Injection | location string into geocode URL | accept | The city name is already percent-encoded by the existing `url_encode` (T-05-WTHR-INJ) before entering the query string; a `lat,lon` is range-checked by `parse_lat_lon`. |
</threat_model>
