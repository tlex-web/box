<objective>
**DESTRUCTIVE — own plan + mandatory adversarial code review.**

- **RENM-V2-02:** `bulk-rename --backup` writes a recoverable pre-rename backup before the rename plan executes; a failed/aborted run leaves the directory recoverable.

Purpose: a pure rename (`std::fs::rename` = `MoveFileExW`) changes only the NAME, so the entire reversible state is the `{old → new}` map — copying bytes protects data that was never at risk (D-22). `--backup` writes a JSON undo MANIFEST (a serde projection of the already-built, collision-cleared `Plan`) to `%LOCALAPPDATA%\box\undo\<id>.json` (OUTSIDE the renamed tree so `--recursive` never re-walks it and it survives renaming the target dir), write+fsync'd in full BEFORE the first rename, then each entry flips `applied:true` as its rename returns — so a mid-batch I/O error leaves a manifest whose `applied` flags exactly partition done-vs-pending and the directory is reconcilable.

Output: `--backup` on `bulk-rename`; new `tests/bulk_rename_backup.rs` with manifest-written, dry-run-no-op, and mid-batch-partition-recoverable tests; a blocking code-review checkpoint. (An automated `--undo` replay subcommand is explicitly Deferred — RENM-V2-02 needs only the backup written + the dir recoverable.)

Depends on 08-03 (shares `src/commands/bulk_rename/mod.rs`; the manifest is a projection of the same build-once `Plan` after 08-03's `{n}`/`--case` transforms and the unchanged pre-flight).
</objective>

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-8-06 | Tampering / data loss | manifest not durable before renames begin | mitigate | Write the FULL all-`applied:false` manifest + `File::sync_all()` BEFORE the first `std::fs::rename`; flip `applied` per rename → an `applied`-partitioned, reconcilable manifest on any mid-batch error (Pitfall 8) |
| T-8-06-LOC | Tampering / data integrity | manifest written inside the renamed tree | mitigate | Write to `%LOCALAPPDATA%\box\undo\<id>.json` (LOCALAPPDATA, not APPDATA), outside the tree; `--recursive` never re-walks it; survives renaming the target dir; fallback to target dir only if LOCALAPPDATA unset |
| T-8-06-ABORT | Tampering / data loss | a rename running before a clean pre-flight | mitigate | Manifest write + renames occur strictly AFTER `preflight_plan` returns clean; the unchanged abort-all-before-any `bail!` writes neither manifest nor renames |
| T-8-06-SILENT | Tampering / data loss | `std::fs::rename` silently overwriting on Windows | mitigate | Inherited: the pre-flight is the entire backstop (no `create_new` for moves); `--backup` is additive recoverability, not a substitute for the pre-flight |
| T-8-06-DOS | Denial of Service | non-UTF-8 NTFS names in the manifest | mitigate | `to_string_lossy()` (D-4) for `old`/`new`; never `to_str().unwrap()` |
| T-8-06-ACCUM | Information Disclosure | manifests accumulate under %LOCALAPPDATA% | accept | No auto-cleanup this phase (`--undo` replay Deferred); documented in RESEARCH § Runtime State Inventory; manifests hold only path names, no secrets |
</threat_model>
