<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| generated script -> user shell | `box completions powershell` output is piped into $PROFILE / Invoke-Expression and EXECUTED in the user's shell |
| install.ps1 -> $PROFILE | the installer can mutate the user's PowerShell profile |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-11-05 | Tampering / Elevation | the generated completion script executed in the user's shell | mitigate | the script is generated by clap_complete from the TRUSTED local `Cli::command()` — no user-controlled string interpolation reaches the output; the only input is the `Shell` enum (clap-validated). The generated-script-runs-in-shell boundary is inherent to all shell completions and is accepted |
| T-11-06 | Tampering | install.ps1 mutating $PROFILE | mitigate | opt-in `-RegisterCompletions` ONLY (no unprompted profile edit — the default path just prints a hint); idempotent `# box completions` sentinel guard (Select-String) prevents duplicate/append loops |
| T-11-07 | Tampering (output purity) | stdout contamination breaking the piped script | mitigate | stdout carries EXACTLY the inert `#` header + the generated script; no stderr chatter (fires even under redirection); no ANSI, no BOM — locked by `powershell_script_is_pure` |
| T-11-08 | Denial of Service | a hostile/huge $PROFILE or generate loop | accept | single-user local CLI; the sentinel guard bounds the append to one block; script size is bounded by the fixed Cli surface |
| T-11-SC | Tampering (supply chain) | new crate `clap_complete` (D-12) | mitigate | Task 1 is a BLOCKING-human legitimacy gate verifying clap_complete is the first-party clap-rs crate on crates.io before the Cargo.toml add; pinned `4.6` to match clap 4.6 (no duplicate clap version) |
</threat_model>
