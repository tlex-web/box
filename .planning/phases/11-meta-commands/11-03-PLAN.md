  `hash.default_algo` arm resolve only `config().hash.default_algo.unwrap_or(Blake3)`
  (config>builtin), while `hash::run_compute` resolves the full
  `cli.or(env).or(config).unwrap_or(builtin)` chain — so with
  `BOX_HASH_DEFAULT_ALGO=md5` and no config, `box hash` emits MD5 but `config
  get`/`show` report `blake3`. The module's own doc comment ("config show can never
  lie about what box hash will consume") is false. Fix: factor a SINGLE `pub(crate)`
  effective-algo resolver in `hash` (env>config>builtin) consumed by BOTH
  `run_compute` (with CLI layered over it) AND config's view, so they cannot drift.
  Weather has no env tier — its arm is untouched.

- **WR-02 — malformed config bricks the repair/inspect/shell-start path (CFG-01 +
  CMP-01).** `main()`'s `init_config()` gate fires for EVERY subcommand before
  dispatch, so a malformed/unknown-key file exits 2 before `box config path`, `box
  config set …` (the intended repair), `box config show`, AND `box completions
  powershell` can run — and an `install.ps1 -RegisterCompletions`-registered
  `$PROFILE` prints the parse error on every new PowerShell session. Fix: decouple
  the config-INDEPENDENT commands (`completions`, and `config path`/`set`) from the
  malformed→exit-2 gate by falling back to a default config ONLY for them; keep
  malformed→exit-2 for every normal command AND for `config show`/`get` (which need
  the effective value). `config set` still round-trips through the same typed
  validate-before-write, so D-03's "self-inflicted lockout is impossible" holds.

Purpose: make `config show`/`get` tell the truth about what `box hash` consumes, and
restore the frictionless-PS7 core value — a corrupt config must be repairable and must
not error every new shell.

Output: an amended `hash`/`config`/`core::config`/`main` set (shared resolver + gate
decoupling) plus WR-01 parity and WR-02 robustness regression tests.
