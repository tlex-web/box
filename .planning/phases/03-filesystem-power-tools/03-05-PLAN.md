<objective>
Ship `box bulk-rename` end-to-end (RENM-01): the phase's single destructive command. It builds a regex rename plan over the target directory's files, runs an in-memory ABORT-ALL-BEFORE-ANY-RENAME pre-flight that detects collisions, cycles/swaps, and path-separator injection, previews the plan by DEFAULT (dry-run; writes nothing), and executes only with `--force`. The entire safety story is the pre-flight check, because `std::fs::rename` SILENTLY OVERWRITES its destination on Windows and has NO `create_new` backstop (D-18, RESEARCH Pitfall 4). Reuses flatten's plan→preview→execute split, `format_row`/`arrow_col` rendering, and the case-folded `occupied`-set model.

Purpose: bulk-rename is the highest-risk command in the phase; it is built last so flatten's dry-run/collision patterns are proven and reused rather than re-derived. The pre-flight detection is LOAD-BEARING — a missed collision is silent data loss.
Output: a working, safe `box bulk-rename` command and Wave-0 integration tests covering every abort path.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| user argv (`pattern`/`replacement`) → bulk-rename | untrusted regex + replacement that drives a DESTRUCTIVE filesystem operation |
| filesystem → bulk-rename | pre-existing on-disk names that a planned target could silently clobber |
| in-memory plan → `std::fs::rename` | the boundary where a missed collision becomes irreversible data loss |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-RENM-clobber | Tampering / DoS (data loss) | `std::fs::rename` silent overwrite (no `create_new` for moves) | mitigate | In-memory ABORT-ALL pre-flight (D-18): every target checked vs other planned targets AND pre-existing on-disk names (case-folded per directory); ANY clobber aborts the whole batch (exit 1, nothing written) in BOTH dry-run and `--force`. The `renm_collision_aborts` test asserts the tree is unchanged on abort. This is the ENTIRE safety story. |
| T-RENM-cycle | Tampering / data loss | swap/cycle `a→b, b→a` | mitigate | Detect-and-abort: any target equal to another item's SOURCE aborts (no silent two-phase temp pass in v1; that is a v2 follow-up). The `renm_cycle_aborts` test asserts the tree is unchanged. |
| T-RENM-pathinject | Tampering (path traversal) | a `${1}` replacement producing `/` or `\` | mitigate | The pre-flight refuses/skips any target containing a path separator (mirrors flatten's `encode_no_separator` invariant); the rename is scoped to the parent dir. The `renm_separator_refused` test enforces it. |
| T-RENM-caseonly | Availability / correctness | case-only rename falsely aborted | accept→mitigate | A case-only change (`foo`→`Foo`) is byte-different and IS a real rename; detected by EXACT non-folded name compare so it is NOT flagged as a self-collision (Pitfall 5). The `renm_case_only_ok` test enforces the valid path. |
| T-RENM-redos | DoS | pathological user regex | accept | The `regex` crate guarantees linear-time matching (no catastrophic backtracking) by construction — inherent mitigation; no extra control needed (RESEARCH Security Domain). |
| T-RENM-panic | DoS | bad regex / unreadable file / deep path | mitigate | `Regex::new` compile error and each `rename` are `.context(...)`-wrapped → clean stderr + exit 1/2, never a panic (FOUND-05, FOUND-06). |
| T-03-SC | Tampering | npm/pip/cargo installs | mitigate | No new dependency is added by this plan (`regex` already present + vetted `[OK]` in RESEARCH.md § Package Legitimacy Audit, rust-lang official). No install-time checkpoint required. |
</threat_model>
