Purpose: The clipboard command that bypasses `clip.exe`'s Unicode mangling and lack-of-paste. The single-shot synchronous main-thread arboard flow satisfies the STATE.md "arboard main-thread only" pitfall AND persists via OS handle ownership (Windows-safe, no keep-alive pump — D-06). The deliberate raw-stdin read (not `read_input`) is the load-bearing detail: routing through `read_input` would inherit the no-arg-TTY→exit-2 contract, which is wrong for a clipboard copy.

Output: A working `box clip` / `box clip --paste` + `tests/clip.rs` + in-module unit tests + Cargo.toml `arboard` dep. `clip` stub gone (2 Phase-5 stubs remain: pomodoro/weather).

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| user/pipe → clip (raw stdin) | Untrusted, possibly non-UTF-8 byte stream crosses into the copy path |
| clip ↔ OS clipboard (Win32) | The user's own clipboard session — read (paste) and write (copy) |
| clipboard → stdout (paste) | Clipboard text re-emitted to a pipe/terminal |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-05-CLIP-DoS | Denial of Service | non-UTF-8 stdin on copy → panic | mitigate | `String::from_utf8(buf).context("clipboard input must be UTF-8")?` → clean exit 1, never `.unwrap()`/panic (FOUND-05). Auto-tested with `\xff\xfe`. |
| T-05-CLIP-THREAD | Tampering / DoS | arboard called off the main thread (STATE.md pitfall) | mitigate | Single-shot synchronous flow: `Clipboard::new()` → one op → return, on the main thread. Never spawn the arboard call onto a worker (D-06). |
| T-05-CLIP-INFO | Information Disclosure | reading/writing the clipboard | accept | Single-user local tool; the clipboard is the user's own session — read/write IS the explicit, requested function (CLIP-01), not a leak. Clipboard contents are NEVER logged (no stderr/stdout echo of the copied text on the copy path). |
| T-05-CLIP-INJ | Tampering (output spoofing) | `--paste` writes clipboard bytes to a pipe/TTY | accept | `--paste` writes the clipboard byte-exact (same posture as `cat`/`xclip`); a consumer rendering arbitrary clipboard bytes to a TTY is out of this tool's scope. No re-color/no-transform path here. |
| T-05-CLIP-SC | Tampering | `arboard` 3.6.1 supply chain | accept | `[VERIFIED: crates.io]` — slopcheck `status: OK` (0 flags), 1Password/tauri-maintained canonical repo, current latest stable (RESEARCH § Package Legitimacy Audit). No blocking legitimacy checkpoint required. |

No elevation is requested or required (D-06). No HIGH-severity threat exists. The genuine control (no-panic on non-UTF-8, main-thread-only arboard) is covered.
